\chapter{Evaluation}

\section{Evaluation Goals}
This chapter evaluates whether the implemented AI collaborator addresses the research questions formulated in the introduction:
\begin{itemize}
    \item Can an integrated, stateful, and supervised agent architecture reduce modeling effort for routine and repetitive tasks?
    \item Can such an architecture lower entry barriers for users with limited optimization expertise?
    \item Can it improve the efficiency and quality of collaborative MILP model development?
\end{itemize}

To answer these questions, we evaluate the system along three dimensions: efficiency, accessibility, and output quality under human supervision.

\section{Experimental Setup}
\subsection{System Configuration}
All experiments are conducted on the integrated prototype described in this thesis, using the same execution pipeline for all tasks and participants. The LLM used for generation is accessed via OpenRouter and configured with the model identifier \texttt{openai/gpt-oss-120b:free}. Each run uses the same system-prompt contract and identical tool availability to ensure configuration consistency across manual-vs-AI comparisons.

The runtime environment consists of three local service endpoints: (i) the modeler web client session, (ii) a LangGraph-backed orchestration runtime for threaded inference and interrupt handling \cite{langchainLangGraph}, and (iii) a browser-automation service using Puppeteer for environment-side model read/write operations \cite{pptrPuppeteerPuppeteer}. AI-generated updates are applied directly to the live collaborative workspace, and all measurements are collected from this end-to-end setup rather than from isolated offline prompts.

\subsection{Task Design}
We use a set of representative MILP modeling tasks with increasing complexity (small, medium, and large). Each task requires users to create or modify variables, objectives, and constraints in the modeler. For each task, two execution modes are compared:
\begin{enumerate}
    \item \textbf{Manual mode:} modeling only through direct UI interaction.
    \item \textbf{AI-assisted mode:} modeling with the first-class AI collaborator through natural-language requests and iterative feedback.
\end{enumerate}

\subsection{Participants and Procedure}
Participants are grouped by experience level (e.g., beginner/intermediate/advanced with optimization modeling). Each participant solves the same task set in both modes. To reduce learning effects, task order should be counter-balanced across participants.

For each run, we log completion time, interaction effort, number of correction cycles, and final model validity. The same acceptance criteria are used for manual and AI-assisted runs.

\section{Metrics}
\subsection{Efficiency Metrics}
\begin{itemize}
    \item \textbf{Task Completion Time (TCT):} end-to-end time from task start to accepted final model.
    \item \textbf{Modeling Speed:} number of completed modeling operations per minute.
    \item \textbf{Speedup Factor:}
    \[
    \text{Speedup} = \frac{T_{\text{manual}}}{T_{\text{AI}}}
    \]
    where values $>1$ indicate faster completion with AI assistance.
\end{itemize}

\subsection{Effort and Accessibility Metrics}
\begin{itemize}
    \item \textbf{Interaction Count:} number of UI operations (manual mode) versus number of prompts/feedback turns (AI mode).
    \item \textbf{Correction Iterations:} number of refinement cycles until user acceptance.
    \item \textbf{Onboarding Performance:} task success rate and completion time for low-experience users.
\end{itemize}

\subsection{Quality and Reliability Metrics}
\begin{itemize}
    \item \textbf{Syntactic Validity Rate:} fraction of generated updates passing structural checks.
    \item \textbf{Accepted Suggestion Ratio:} accepted proposals divided by total AI proposals.
    \item \textbf{Final Model Quality:} whether resulting models satisfy task requirements and solver feasibility conditions.
\end{itemize}

\section{Threats to Validity}
\subsection{Internal Validity}
Learning effects and prompt-quality variance may influence measured performance. Counter-balanced task ordering and standardized task instructions mitigate these effects.

\subsection{External Validity}
Results from prototype tasks and a limited participant set may not generalize to all optimization domains. Future studies should include broader industrial case studies and larger user cohorts.

\subsection{Construct Validity}
Time-based metrics alone cannot capture all benefits of collaboration quality. Therefore, efficiency metrics are complemented by validity, acceptance, and refinement metrics.

\section{Summary}
This evaluation framework directly addresses all three research questions by combining quantitative efficiency comparisons (manual versus AI-assisted), accessibility indicators for less-experienced users, and quality/reliability metrics for collaborative HITL modeling. The resulting evidence determines whether the implemented first-class AI collaborator improves modeling speed, reduces entry barriers, and increases collaborative model-development quality.
